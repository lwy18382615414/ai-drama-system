# 前端流程设计(小说 → 短剧 工作台)

> 本文基于重构后的后端架构(见 `AI漫剧后端架构优化重构方案.md`),定义 Web 工作台的信息架构、
> 核心交互、状态同步机制与实施顺序。技术栈沿用现有 Vue 3 / Vite / Naive UI。
> 后端契约引用:派生管线状态、`generation_jobs`、revision/stale 治理、成本记录、SSE 任务流。

## 1. 文档目标

重构后的后端向前端暴露四类新能力:

1. **派生式管线状态**:每集六阶段 × 六种展示状态,由后端统一计算;
2. **Job 级批量进度**:父子任务聚合,支持取消与失败项重试;
3. **revision / stale 治理**:上游修改精确标记下游过期,重规划有影响预览;
4. **成本记录与预估**:批量操作前预估,项目级累计费用可查。

前端设计的核心任务是把这四类能力转化为用户可理解的交互,而不是把任务表和状态字段直接搬上界面。

## 2. 设计原则

### 2.1 后端是唯一真相

前端**永远不自行推导管线状态**。"这一集能否生成分镜"、"这张图是否过期"全部以
`computeEpisodePipelineStatus()` 的返回为准,前端只负责渲染与按钮门控(置灰 + 展示后端给出的原因)。
禁止在前端出现"有剧本 → 所以可以抽资产"这类第二套推导逻辑。

### 2.2 事件只触发刷新,不承载数据

SSE 事件不直接写入前端状态,而是触发对应 query 的失效重取(invalidate)。数据始终以 REST
响应为准。这与后端"snapshot 对齐、事件催更"的设计同构,断线重连后天然一致,无需前端做状态合并。

### 2.3 乐观更新仅限纯编辑

剧本文本、分镜 prompt 等纯编辑操作可做乐观更新;**生成类操作一律等待真实状态**。
生成没有"乐观"可言,假进度比慢更伤信任。

### 2.4 stale 是状态,不是错误

过期内容照常展示与浏览,只阻断下游生成。黄色角标 + 过期原因 + "重新生成"入口,
不弹错误、不清空内容。

## 3. 状态架构与技术选型

| 关注点 | 选型 |
|--------|------|
| 服务端状态 | TanStack Query(`@tanstack/vue-query`):缓存、失效、重取、轮询降级 |
| UI 状态 | Pinia:仅存纯界面状态(面板开合、多选集合、向导步骤) |
| 实时同步 | 单一 EventSource(每项目一条),封装为 `useProjectTaskStream(projectId)` |
| 组件库 | Naive UI(沿用) |

### 3.1 Query Key 设计

Query key 与后端资源一一对应,失效粒度即由此确定:

```text
['projects']
['project', projectId]
['episode-pipeline', projectId]          # 全项目管线看板(批量接口)
['episodes', projectId]
['script', episodeId]
['episode-assets', episodeId]
['storyboards', episodeId]
['characters', projectId]
['appearance-versions', characterId]
['jobs', projectId]                      # 活动 Job 列表
['job', jobId]
['usage', projectId]                     # 费用聚合
```

### 3.2 SSE → invalidate 映射

`useProjectTaskStream` 内部维护事件类型到 query key 的映射,示例:

| 事件 | invalidate |
|------|-----------|
| task 认领/结算(script_generation) | `['script', episodeId]`、`['episode-pipeline', projectId]` |
| task 结算(image_generation) | 对应目标资源 key、`['episode-pipeline', projectId]` |
| job 进度更新 | `['job', jobId]`、`['jobs', projectId]` |
| stale 传播事件(如后端推送) | `['episode-pipeline', projectId]` + 受影响资源 key |
| 连接建立收到 snapshot | 全量对齐活动任务视图,invalidate `['jobs', projectId]` |

### 3.3 降级策略

SSE 断开时:任务抽屉与看板降级为 10s 轮询(`refetchInterval`),顶部横幅提示
"实时连接已断开,正在轮询"。重连成功后收 snapshot 对齐并撤掉横幅。

## 4. 信息架构

```text
项目列表
  └── 项目工作台 /projects/:projectId
        ├── 章节与事件      /chapters      导入、事件抽取状态
        ├── 分批规划        /batches       批次列表、重规划向导入口
        ├── 集列表(看板)  /episodes      管线看板 —— 核心页面
        ├── 资产库          /assets        角色/场景/道具、形象版本管理
        └── 设置与费用      /settings      累计成本、任务历史、项目配置
  └── 集工作区 /episodes/:episodeId
        ├── 剧本    tab
        ├── 资产    tab(本集连接的角色/场景/道具)
        ├── 分镜    tab
        └── 图片    tab(参考图 + 首帧总览)
全局层
  └── 任务中心(顶栏指示器 + 抽屉,任意页面可达)
```

## 5. 核心页面设计

### 5.1 集列表:管线看板(主轴页面)

每集一行,横向六个阶段格:**事件 → 规划 → 剧本 → 资产 → 分镜 → 图片**。
格子渲染派生状态:

| 派生状态 | 视觉 | 交互 |
|----------|------|------|
| `not_started` | 灰 | 若依赖就绪,悬浮显示"生成"入口 |
| `queued` | 蓝 · 静态 | 显示排队位置(可选) |
| `running` | 蓝 · 转圈 | 点击展开任务详情,可取消 |
| `ready` | 绿 | 点击跳转集工作区对应 tab |
| `stale` | **黄** | 角标 + 悬浮显示过期原因,点击进入重新生成流程 |
| `failed` | 红 | 悬浮显示错误分类文案 + 对应动作(见 §7) |

看板特性:

- 数据来自批量 pipeline status 接口,**一次请求整页**,禁止逐集请求;
- 多选集 → 底部浮出批量操作栏("为选中 5 集生成剧本"等)→ 走成本确认(§8)→ 创建 Job;
- 数百集时启用虚拟滚动;
- 顶部汇总条:活动 Job 数、stale 项总数("本项目有 12 项过期内容 → 查看")。

### 5.2 集工作区

四个 tab 共享同一份该集的 pipeline status。关键交互:

- **依赖门控**:未就绪阶段的生成按钮置灰,tooltip 显示后端返回的原因
  ("需要先生成剧本")。用户被界面引导走完管线,无需文档。
- **剧本 tab**:编辑器 + 保存。保存成功反馈中直接告知失效结果:
  "已保存。资产、分镜和 4 张图片已标记过期",附 [稍后处理] / [立即重新生成] 两个动作。
  不静默标记。
- **分镜 tab**:镜头卡片列表,含 image_prompt 编辑(乐观更新)、首帧图预览、
  单镜头重生成按钮;生成中的卡片显示骨架占位 + 单任务进度。
- **图片 tab**:整集图像总览(角色参考图 / 场景参考图 / 首帧),
  "一键生成全部"入口 → 成本确认 → Job;页面顶部横幅显示进行中 Job 的进度条。
- 存在任何 stale 时,工作区顶部显示汇总条:
  "本集有 3 项内容因剧本修改而过期 → [查看] [全部重新生成]"。

### 5.3 全局任务中心

- 顶栏任务指示器(类似 IDE 后台任务图标),有活动 Job 时显示计数与微型进度环;
- 点开为抽屉,按 Job 分组:"第 3 集全部图片生成 12/18,2 失败",
  每项提供 [取消 Job] [重试失败项] [展开子任务];
- 已结算 Job 保留最近若干条,可跳转到相关集;
- 数据源:`['jobs', projectId]` + SSE job 事件。

用户发起批量后可以离开当前页面去做别的——任务抽屉让异步系统的价值对用户成立。

### 5.4 资产库与形象版本

- 角色/场景/道具三个列表,项目级复用视角;
- 角色详情内嵌**形象版本时间轴**:自动版本标注来源集、手动版本标注生效锚点,
  每版带参考图缩略图与重生成入口;
- 新增手动版本表单:生效锚点选择器**优先选剧情事件**(`novel_events`,不参与重编号),
  集号作为回退选项——与后端重构结论一致;
- 角色描述修改后,受影响的参考图与首帧按 §6 规则显示 stale。

### 5.5 设置与费用

- 累计费用卡片:按类型分列(文本 token / 图片张数 / 费用),数据来自 `usage_records` 聚合;
- 按集 / 按任务类型的费用明细表,用于发现异常调用;
- 任务历史列表(含失败任务与错误分类),支持按类型筛选。

## 6. stale / revision 交互流程

### 6.1 呈现规则

- stale 内容**不阻断浏览**,只阻断下游生成;
- 黄色角标必须附带原因:"已过期(上游剧本已修改)"——原因来自失效传播记录;
- 卡片主按钮从"生成"变为"重新生成";
- 汇总条聚合展示,提供批量重生成入口。

### 6.2 重规划向导(最重的确认流程)

三步向导,禁止一键直达:

```text
第 1 步  选择批次与规划参数
第 2 步  影响预览(后端返回):
         - 将替换 N 集
         - 其中 M 集剧本含手工修改(单独标红列出 —— 用户最心疼的东西)
         - 已生成图片 X 张将被标记为已取代
         - 重新生成预估成本 ¥Y
第 3 步  勾选"我了解旧内容将被标记为已取代"后,提交按钮才可用
```

提交后旧数据标记 `superseded`(后端行为),前端在批次页提供只读的
"查看历史版本"入口(第一版可不做,不建 diff 界面——与后端结论一致:保留数据、功能等真实需求)。

### 6.3 剧本编辑闭环

```text
用户保存剧本 → 后端标记下游 stale → 保存成功反馈中列出失效清单
→ [稍后处理](看板对应格子变黄) / [立即重新生成](逐项或整链入队,走成本确认)
```

## 7. 错误分类 → 用户文案与动作映射

前端维护一张映射表,禁止把原始错误直接抛给用户:

| `error_code` | 文案 | 可用动作 |
|--------------|------|----------|
| `content_rejected` | 内容被审核拒绝,请修改描述/提示词后重试 | [编辑后重试](**不提供**自动重试) |
| `rate_limited` | 服务繁忙,将自动重试 | 无需操作,显示下次重试倒计时(如有 `next_retry_at`) |
| `provider_timeout` | 生成超时,将自动重试 | 同上 |
| `validation_error` | 生成结果格式异常,已自动修复失败 | [重试] |
| `configuration_error` | 服务配置异常 | 引导检查 Provider 配置(单人部署即本人) |
| `dependency_not_ready` | 上游内容未就绪 | 跳转到对应上游阶段 |
| `internal_error` | 系统异常,已记录 | [重试] |

## 8. 成本确认:两道闸门

1. **提交前**:所有批量操作(整集图片、多集剧本、重规划重生成)弹确认框,
   展示后端预估:"将生成 18 张图片,预估 ¥2.7"。单目标小操作不弹,避免交互变碎;
2. **事后可查**:设置与费用页的累计费用与明细(§5.5)。

确认框中的预估金额**必须来自后端接口**,前端不做单价计算。

## 9. 关键 Composable 与组件清单


## 10. 分阶段实施(对齐后端阶段)

| 前端阶段 | 依赖后端 | 内容 |
|----------|----------|------|
| F1 | 后端阶段一 | 引入 vue-query 与 SSE composable;错误分类文案映射;任务取消按钮;成本记录展示(费用页只读) |
| F2 | 后端阶段二 | stale 角标与汇总条;剧本编辑闭环;重规划三步向导;形象版本锚点选择器改造 |
| F3 | 后端阶段三 | 管线看板(替代现有集列表);全局任务中心;批量操作 + 成本确认;Job 进度替代逐任务轮询 |

F1 可与后端阶段一并行开发(接口契约先行);F3 的看板是一次页面级替换,建议灰度:
旧集列表与新看板并存一个迭代,确认后下线旧页。

## 11. 本轮不做的内容

- revision diff 比较界面与完整回滚 UI(保留只读历史入口即可);
- 多用户相关界面(登录、成员、配额)——随后端"开放注册前"节奏;
- 移动端适配(工作台形态以桌面为先);
- 前端本地缓存持久化(vue-query 内存缓存足够,避免引入一致性问题);
- 视频阶段的任何 UI 占位。

## 12. 验收标准

1. 任意页面刷新或 SSE 断线重连后,界面状态与数据库一致,无残留假进度;
2. 依赖未就绪的生成按钮全部置灰且能看到原因,不存在可点击但必然 422 的入口;
3. 上游修改后 3 秒内看板对应格子变黄,且用户在保存反馈中已被告知;
4. 重规划必须经过影响预览与显式勾选,无旁路;
5. 批量生成数百任务时页面不卡顿(虚拟滚动 + 批量接口),进度以 Job 聚合展示;
6. 所有失败任务展示分类文案与正确动作,无原始错误串直出;
7. 任何批量操作前用户都见过预估成本。
